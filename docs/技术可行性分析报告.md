# 技术可行性分析报告

> 项目: EmberIntel Serial Monitor v2.0  
> 日期: 2026-05-17  
> 版本: 1.0  

---

## 1. 项目背景

将现有 Python CLI 串口监控工具全面重写为 C++/Qt5 桌面应用，实现：
- GUI 桌面界面（开发者日常使用）
- CLI 命令行界面（AI Agent / 脚本使用）
- 双进程 IPC 架构（GUI 常驻保活串口，CLI 随时连接）

---

## 2. 技术栈可行性分析

### 2.1 C++17 + Qt5

| 评估维度 | 分析 | 结论 |
|----------|------|------|
| 串口支持 | QSerialPort 是 Qt5 内置模块，API 成熟稳定 | ✅ 可行 |
| GUI 能力 | Qt5 Widgets 提供完整的桌面控件体系 | ✅ 可行 |
| IPC 通信 | QLocalSocket/QLocalServer 内置跨平台本地通信 | ✅ 可行 |
| JSON 处理 | rapidjson 提供极致性能的 SAX/DOM API | ✅ 可行 |
| 跨平台 | Qt5 支持 Windows/Linux/macOS | ✅ 可行 |
| 打包部署 | windeployqt 自动收集依赖 DLL | ✅ 可行 |

### 2.2 关键技术风险与对策

| 风险 | 等级 | 对策 |
|------|------|------|
| QSerialPort 高波特率稳定性 | 低 | Qt5 经过多年验证，3Mbps 无压力 |
| IPC 大数据量传输延迟 | 低 | QLocalSocket 基于 Named Pipe，本地延迟 < 1ms |
| 多串口并发读取性能 | 中 | 每个串口独立 QThread，避免主线程阻塞 |
| rapidjson API 复杂度 | 低 | 仅用于配置读写和日志导出，场景简单 |
| Qt5 与 Qt6 未来迁移 | 低 | Qt5 Widgets API 在 Qt6 中基本兼容 |

### 2.3 依赖库清单

| 库 | 版本 | 用途 | 许可证 |
|----|------|------|--------|

## 3. 架构可行性分析

### 3.1 双进程 IPC 架构

```
┌─────────────────┐         QLocalSocket          ┌─────────────────┐
│   GUI Process   │◄══════════════════════════════►│   CLI Process   │
│   (Server)      │       JSON over Named Pipe     │   (Client)      │
│                 │                                │                 │
│  常驻后台        │                                │  按需启动/关闭   │
│  持有串口连接    │                                │  无串口依赖      │
│  管理日志缓冲    │                                │  轻量级         │
│  提供IPC服务     │                                │  脚本友好        │
└─────────────────┘                                └─────────────────┘
```

**可行性验证：**

| 场景 | 预期行为 | 可行性 |
|------|----------|--------|
| CLI 启动时 GUI 未运行 | CLI 提示 "GUI 服务未运行" 并退出 | ✅ |
| CLI 连接后获取实时日志 | GUI 通过 IPC 推送日志流 | ✅ |
| CLI 发送串口数据 | CLI → IPC → GUI → QSerialPort | ✅ |
| CLI 关闭 | GUI 继续运行，串口数据不丢失 | ✅ |
| 多个 CLI 同时连接 | GUI 广播日志到所有 CLI 客户端 | ✅ |
| GUI 关闭 | 通知所有 CLI 客户端断开 | ✅ |

### 3.2 多串口并发

每个串口连接运行在独立 `QThread` 中：
- 读取线程：非阻塞轮询 `QSerialPort::readyRead` 信号
- 主线程：UI 渲染 + IPC 服务
- 线程安全：日志缓冲区使用 `QMutex` 保护

## 4. 性能可行性分析

### 4.1 关键性能指标

| 指标 | 目标 | 实现方式 |
|------|------|----------|
| 串口读取延迟 | < 10ms | QSerialPort 异步信号，非阻塞读取 |
| 日志缓冲区追加 | O(1) | `std::deque` + `QMutex` |
| GUI 日志渲染 | 10000行不卡顿 | 虚拟列表，仅渲染可见行 |
| IPC 日志推送延迟 | < 5ms | QLocalSocket 本地 Named Pipe |
| 内存占用 | < 100MB | 10000行 × 256字节 ≈ 2.5MB 缓冲区 |

### 4.2 压力场景

| 场景 | 数据量 | 预期表现 |
|------|--------|----------|
| 115200bps 持续输出 | ~11.5KB/s | 无压力 |
| 921600bps 持续输出 | ~92KB/s | 轻微 CPU 占用 |
| 4 个串口同时 115200bps | ~46KB/s | 正常 |
| 10000 行日志缓冲区满 | 循环覆盖 | O(1) 操作 |

## 6. 结论

**技术方案完全可行。** C++17 + Qt5 + CMake 技术栈成熟稳定，所有核心功能（串口通信、GUI、IPC、JSON）均有成熟库支持，无技术阻塞点。双进程 IPC 架构有效解决了 CLI 关闭导致串口数据丢失的核心痛点。
